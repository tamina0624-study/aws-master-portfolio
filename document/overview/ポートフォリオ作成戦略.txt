🏗️ 題材1：ハイブリッド・クラウド接続とセキュリティ自動化
【ターゲット：大手SIer（日立ソリューションズ、日本IBM等）】
オンプレミス（物理環境）とAWSを繋ぐ、実務で最も需要のある構成です。

内容: 自宅のルーター（Yamaha等）とAWSをL2TP/IPsecで接続し、AWS側の環境をセキュアに構築します。

見せ場:

ネットワーク: AWS Client VPNやSite-to-Site VPNの設定。

セキュリティ: GuardDutyで検知した脅威を、Lambda経由でAWS WAFに自動反映し、攻撃IPをブロックする「自律防御システム」。

解説ポイント: 「なぜ物理ルーターとの接続にこだわったか」を語ることで、ハードウェアとクラウド両方の知識をアピール。


🚀 題材2：EKS（Kubernetes）上でのDevSecOpsパイプライン
【ターゲット：外資コンサル、メガベンチャー（アクセンチュア等）】
「全冠（All Cert）」を目指す姿勢と、モダンな開発運用スキルをアピールします。

内容: Reactで作成したシンプルなアプリを、EKS（Kubernetes）上で稼働させます。

見せ場:

コンテナ管理: HelmチャートによるKubernetesリソースの管理。

CI/CD: GitHub Actionsを使い、コードをpushしたら自動で脆弱性診断（Trivy等）が走り、EKSへデプロイされる仕組み。

監視: PrometheusとGrafanaをデプロイし、リソース使用率を可視化。

解説ポイント: 「インフラエンジニアがアプリのライフサイクルをどう支えるか」という視点を強調。


🔐 題材3：AWS Security Specialty 実装特化型テンプレート
【ターゲット：セキュリティエンジニア、海外副業】
「SCS合格」の知識を、目に見える「成果物」としてパッケージ化します。

内容: 「これ一つでAWSのベストプラクティスに則った初期環境が完成する」というTerraformテンプレートの作成。

見せ場:

権限管理: IAM Roleの最小権限設定（PoLP）を徹底したコード。

ログ管理: CloudTrailとConfigを有効化し、S3に暗号化（KMS）して保存、Athenaで分析できる状態にする。

自動修復: 「パブリック公開されたS3バケットを検知したら、即座に非公開に戻す」イベント駆動の設定。

解説ポイント: 海外副業を意識し、READMEを英語で完璧に整え、「再利用性」の高さをアピール。

🛠️ 実務直結型ポートフォリオ：
「エンタープライズ基準のAWSランディングゾーンと運用自動化基盤」
この構成は、単一のアプリ用インフラではなく、**「企業がAWSを利用し始める際に最初に作る、堅牢な共通基盤」**を想定しています。

1. ネットワーク：単なるVPNではなく「ハイブリッド接続の冗長化」
実務ではVPNが切れることは許されません。

実務構成: AWS Site-to-Site VPNを、あえて**「冗長構成（2本のトンネル）」**で構築し、BGPによる動的ルーティングを設定します。

ポイント: Yamahaルーター側とAWS側で、障害時にどう切り替わるか（フェイルオーバー）の検証結果をドキュメント化します。これが「ネットワークエンジニア」としての深みになります。

2. セキュリティ：単なる検知ではなく「統制（ガードレール）」
実務では、個別の設定よりも「誰が何をやっても安全な仕組み」が求められます。

実務構成: AWS Control Tower または Organization を意識したマルチアカウント構成のシミュレーション。

実装要素: * Service Control Policies (SCP): 「特定のリージョン以外は使わせない」「特定の強力な操作を禁止する」といった組織レベルの制限。

CloudConfigによる自動修正: 「パブリック公開されたS3を検知して自動で閉じる」といった修復コードの実装。

3. ID・アクセス管理：単なるIAMではなく「SSOと最小権限のライフサイクル」
実務構成: AWS IAM Identity Center (旧SSO) を活用。

ポイント: 開発者・運用者・監査者といった「ロール」ごとに、権限をどう切り分けたか（職務分掌）をTerraformで定義します。

4. 運用・監視：単なるダッシュボードではなく「アラートの運用設計」
実務で最も苦労するのは「大量のアラートへの対応」です。

実務構成: CloudWatchアラートを、重要度（High/Low）によって通知先を分ける仕組み。

ポイント: Slack通知だけでなく、**「システム構成図の何番の箇所の異常か」**が即座にわかるようなメッセージ整形をLambdaで行います。

1. ネットワーク：VPNの「MTUサイズ」と「ルーティング」の罠
これは、ネットワークエンジニアとしてのバックグラウンドとAWSを繋ぐ際に、最も「実務の匂い」がするポイントです。

躓きポイント: * VPN越しだと特定のWebサイトや大きなファイル転送だけが失敗する（MTU/MSS問題）。

オンプレミス側のセグメントが増えた際、動的ルーティング（BGP）が適切に伝播せず、一部の通信がブラックホール化する。

ポートフォリオへの盛り込み: * 「あえて」BGPを用いた冗長化構成にし、特定のルートを手動で制御する（AS Path Prepend等）設定を含める。

トラブルシューティングとして「TCP MSS調整」をルーター側で行った経緯を解説する。

アピール: 「クラウドだけでなく、パケットレベルで物理ネットワークの挙動を理解している」ことを証明。



2. セキュリティ：IAMの「循環参照」と「境界（Boundaries）」
SCS（Security Specialty）で学ぶ知識が、実務で牙を剥くパターンです。

躓きポイント: * 「最小権限（PoLP）」を攻めすぎて、AWSサービス同士の連携（例：LambdaからS3への書き込み）が、KMSの権限不足で失敗する。

IAMポリシーの文字数制限に達してしまい、複雑な権限管理が破綻する。

ポートフォリオへの盛り込み: * Permissions Boundary（権限の境界） を使用して、開発者に権限委譲しつつも、特権昇格できない「ガードレール」を構築する。

KMSのキーポリシーとIAMポリシーの両方で適切に許可を与えないと動かない、という「二重の鍵」の構造を解説する。

アピール: 「単に許可を与えるだけでなく、組織的な統制と運用性を両立できる」視点を証明。


3. Kubernetes (EKS)：コンテナの「起動順序」と「外部公開の壁」
Kubernetesを実務で導入して最初に誰もが頭を抱えるポイントです。

躓きポイント: * DBが起動する前にアプリが起動してクラッシュループに陥る。

ALB（Application Load Balancer）との連携において、ターゲットグループがいつまでも Unhealthy から変わらない。

ポートフォリオへの盛り込み: * Init Containers を使ってDBの起動を待機する仕組みを入れる。

AWS Load Balancer Controller の導入過程で、サブネットのタグ付け忘れや、セキュリティグループの不備によって通信が疎通しなかった「失敗と解決」を解説。

アピール: 「抽象化されたコンテナ技術の下で動いている、具体的なAWSリソースの挙動を制御できる」ことを証明。

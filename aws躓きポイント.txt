1. ネットワーク（AWS × オンプレ連携）
インフラ出身者がAWSを触る際、最も「物理と仮想のギャップ」に苦しむ領域です。

非対称ルーティングの罠:

現象: 行きと帰りのパケット経路が異なり、ステートフルなファイアウォール（FortiGate等）でドロップされる。

躓き: AWSの複数VPNトンネルやDirect Connectの冗長化時にBGPの優先順位（AS Path Prepend）を間違えると発生。

TCP MSS調整（MTU問題）:

現象: pingは通るが、Webページ（大きなパケット）が開かない。

躓き: カプセル化によるオーバーヘッドを考慮せず、ルーターでMSS書き換えの設定を忘れる。

Security Group と Network ACL の使い分けミス:

現象: 通信が通らない。

躓き: ACLが「ステートレス」であることを失念し、戻りのパケットを許可し忘れる。

2. セキュリティ（IAM & データ保護）
SCS（Security Specialty）の試験には出ますが、実装で必ずハマるポイントです。

KMSキーポリシーの閉め出し:

現象: 管理者（Root）ですら鍵を管理できなくなる。

躓き: キーポリシーで Enable IAM User Permissions を入れ忘れ、特定のロールのみを許可してそのロールを削除してしまう。

S3バケットポリシーとIAMの競合:

現象: IAMで許可しているのに 403 Forbidden になる。

躓き: バケットポリシー側に明示的な Deny があり、それがIAMの Allow を上書きしている（AWSの評価論理の理解不足）。

クロスアカウントの権限継承:

現象: アカウントAのLambdaが、アカウントBのS3にあるファイルを読み取れない。

躓き: リソース側のポリシー（S3）と、実行側の権限（IAM Role）の両方に許可が必要なことに気づかない。

3. Terraform (IaC)
コードで管理し始めた瞬間に直面する、運用の壁です。

Stateファイルの競合と破損:

現象: ローカルで実行してStateファイルがズレ、本番環境を壊しそうになる。

躓き: S3でのState管理とDynamoDBによる排他制御（Locking）を導入していない。

循環参照（Circular Dependency）:

現象: terraform apply が「Aを作るにはBが必要、Bを作るにはAが必要」とエラーを出す。

躓き: セキュリティグループ同士がお互いをソースとして参照し合う設定など。

意図しないリソースの強制置換:

現象: 軽微な設定変更のつもりが、DBインスタンスが「削除→再作成」されてデータが消えそうになる。

躓き: どのパラメータが Forces replacement（再作成を伴う）になるかを把握せずに apply する。

4. Kubernetes (EKS)
最も「ハマりどころ」が多い、モダンインフラの最難関です。

ALB Ingress Controllerの疎通不良:

現象: Ingressを作っても一向にALBが立ち上がらない、または Unhealthy になる。

躓き: サブネットへの特定のタグ（kubernetes.io/role/elb）付け忘れ。

PodのIPアドレス枯渇:

現象: ノードに余裕があるのに、新しいPodが Pending になる。

躓き: AWS VPC CNIの仕様により、インスタンスタイプごとに持てるIP数が決まっていることを考慮していない。

ゾンビプロセスとGraceful Shutdown:

現象: デプロイ（入れ替え）時に通信が一瞬途切れる、または処理中のデータが消える。

躓き: アプリが SIGTERM を正しく受け取らず、強制終了されている（preStopフックの未設定）。

という語り口をすると、エンジニアとしての説得力が一気に増します。まずは第1週のTerraform学習で、**「State管理の失敗」**あたりをあえて経験してみるのも面白いかもしれません。






1. ネットワーク・ハイブリッド構成（接続の「質」と「運用」）
単に繋がるだけでなく、運用フェーズで発覚する問題です。

VPNトンネルの「片肺」問題とデッドピア検知 (DPD):

現象: 片方のトンネルがダウンした際、切り替わりに数分かかる、あるいは通信が戻ってこない。

躓き: DPD（死活監視）の間隔設定がAWS側とルーター側で合っておらず、ゾンビセッションが残ってしまう。

DNS名前解決の「再帰」ループ:

現象: オンプレからAWSのリソース（RDS等）を名前で引けない、またはその逆。

躓き: Route 53 Resolver（インバウンド/アウトバウンドエンドポイント）の設定で、条件付きフォワーダーがループ設定になり、クエリが無限回廊に陥る。

NAT Gatewayのポート枯渇:

現象: 通信が断続的にエラーになる。

躓き: 大量の外部APIを叩くようなシステムで、NAT Gatewayのソースポート（65535制限）が枯渇することを見落とす。

2. セキュリティ・ガバナンス（実務の「制約」）
「設定したはず」が「なぜか効かない」という、ポリシープライオリティの罠です。

OrganizationsにおけるSCPの「継承」ミス:

現象: 子アカウントでフル権限（Administrator）を持っているのに、特定の操作だけ「Access Denied」になる。

躓き: 親アカウントで設定したSCP（サービス制御ポリシー）が、子アカウントのIAM権限よりも優先される（強い拒否）ことを忘れている。

S3の「パブリックアクセスブロック」とポリシーの矛盾:

現象: バケットポリシーで公開設定にしても公開されない。

躓き: アカウントレベルの「ブロックパブリックアクセス」設定がオンになっており、個別のポリシー設定が無視されている。

VPCエンドポイントポリシーの「絞りすぎ」:

現象: VPC内からのAWSサービス（S3等）へのアクセスが急に失敗する。

躓き: エンドポイントにポリシーを付けた際、特定のロール以外の通信を意図せず全て遮断してしまう。

3. Terraform (IaCの「ライフサイクル」)
構築時ではなく、2回目以降の実行や環境変更で発生する問題です。

「Drift（ドリフト）」による意図しない上書き:

現象: 誰かがAWSコンソールで手動変更した設定が、terraform apply で元に戻ってしまい、サービスが停止する。

躓き: 運用フェーズでコンソール操作を許容してしまい、コードと実態が乖離（Drift）する。

Providerバージョンの不一致:

現象: 同じコードなのに、別のPCで実行するとエラーになる。

躓き: required_providers でバージョンを固定しておらず、最新版の破壊的変更を食らう。

機密情報の「State露出」:

現象: DBのパスワードなどがStateファイルに平文で記録されてしまう。

躓き: TerraformのStateファイル自体を適切に暗号化（S3のKMS暗号化等）し忘れる。

4. EKS / コンテナ（リソースとコストの「現実」）
クラウドならではの「動的な挙動」によるトラブルです。

CoreDNSのスケール不足:

現象: アプリの負荷が上がると、急にDB接続エラー（名前解決失敗）が多発する。

躓き: クラスター標準のDNSサービス（CoreDNS）のリソース割り当てが初期設定のままで、DNSクエリのスパイクに耐えられない。

StorageClassとアベイラビリティゾーン (AZ) の制約:

現象: Podを再起動したら、EBS（ディスク）がマウントできずに起動失敗する。

躓き: EBSは特定のAZに固定されるため、Podが別のAZのノードで起動しようとすると、ディスクに手が届かなくなる（実務ではEFSやAZ指定が必要）。

ImagePullBackOff（プライベートレジストリ）:

現象: コンテナイメージが取得できず起動しない。

躓き: ECRへのアクセス権限（IAM）はあるが、VPCエンドポイントが未設定、またはNAT Gateway経由のルートがなく、ノードが外にイメージを取りに行けない。
